## Внутренние и внешние сервисы

Если в инструкции приводится пример или выборка, необходимо строго следовать указанным формулировкам. Текст должен быть написан точно так, как указано в инструкции.
1. Полное название
Название/бренд продукта (которое используется при любых коммуникациях касательно данного продукта)
* Правила именования сервисов 
Краткое название (код) должно быть уникальным и формироваться по следующему принципу:
<продукт*>-<подпродукт>-<*globalITservice>-<датацентр>
где <globalITservice> - глобальное решение используемое для сервиса. В данном поле указывается общеизвестный тип решения, например: apache, pgsql, tomcat, zabbix.
Датацентр используется только в случае, если одинаковые сервисы находятся в разных ЦОДах.
Если решение самописное, необходимо указывать его внутреннее название в команде.
Если какая-то составная часть названия не применима к сервису, ее можно опустить.
Полное название формируется из названия глобального решения и для какого продукта/команды оно используется, возможно с указанием других уникальных атрибутов (например DC)

2. Описание

Бизнесовое описание продукта, из которого будет понятно:
* назначение
* целевая аудитория
* продукт для внешних пользователей или для сотрудников
* ключевые бизнес-процессы компании на которые влияет данный продукт
* окружение dev/stage/prod

3. Домены

Домены зарегистрированные и использующиеся данным продуктом, включая те с которыми пользователи не взаимодействуют напрямую, но которые используются для функционирования продукта.

4. Ссылка на Wiki

Ссылка на внутреннюю документацию.

5. Проект в таск-трекере

Проект в таск-трекере (напр. Youtrack) в которых команда ведет тикеты/задачи на разработку.

6. Владелец
* Контакт ответственного за продукт (CPO, PO, PM, Team Lead, QA Lead, Architect...)
* Профиль сотрудника на team.wb.ru - EID

(обязательна для инфраструктурного сервиса) - указывается ID команды, которая будет поддерживать/администрировать данный сервис.

(обязательна для бизнес-сервиса) - указывается ID команды,  которая занимается развитием или разработкой данного сервиса.

9. Ответственный за ИБ
